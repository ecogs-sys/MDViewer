# MDViewer — Design Spec

**Date:** 2026-05-16
**Status:** Approved

---

## Overview

MDViewer is a cross-platform desktop application for browsing and reading markdown files generated by AI agentic workflows. It is a read-only viewer — no editing, no authoring. Users open a folder, navigate a file tree, and read rendered markdown with full formatting support including Mermaid diagrams, images, math, tables, syntax-highlighted code, frontmatter, and callouts.

---

## Goals

- Launch from a desktop/taskbar shortcut with no terminal or configuration required
- Browse all `.md` files in a selected folder and its subfolders via a collapsible file tree
- Render markdown fully: Mermaid diagrams, images, GFM tables, LaTeX/math, syntax-highlighted code blocks, YAML frontmatter panel, and GitHub-style callouts
- Support dark/light theme that follows OS setting by default, with a manual override toggle
- Remember the last opened folder across sessions so re-launch is instant
- Fuzzy search across all markdown files in the open folder (nice-to-have)

## Non-Goals

- File editing or annotation
- Git integration
- Custom AI workflow conventions or special folder structure parsing
- Cloud sync or remote file access

## Architecture

### Process Model

Electron splits into two isolated processes that communicate via IPC:

**Main Process (Node.js)**
- Full access to the file system and native OS APIs
- Manages app lifecycle, BrowserWindow creation, and the native menu bar
- Exposes file system operations to the renderer exclusively through named IPC channels
- Runs `electron-store` for preferences persistence

**Renderer Process (React + TypeScript)**
- Sandboxed browser context — no direct file system access
- All file operations go through IPC calls to the main process
- Runs the entire React UI and the unified markdown pipeline

### IPC Channels

Defined as constants in `src/shared/types.ts` to keep the contract explicit:

| Channel | Direction | Payload |
|---|---|---|
| `dialog:open-folder` | renderer → main → renderer | returns `string` (folder path) |
| `fs:read-tree` | renderer → main → renderer | sends path, returns `FileNode` tree |
| `fs:read-file` | renderer → main → renderer | sends path, returns raw markdown `string` |
| `fs:watch-start` | renderer → main | sends path, starts fs watcher |
| `fs:watch-event` | main → renderer | fires on file system change; renderer refreshes tree |

The file watcher means the tree and preview stay current if an agent writes new files while the app is open — no manual reload needed.

### Launch Flow

**First launch:**
1. App opens → checks `electron-store` for saved folder path → none found
2. Renders `WelcomeScreen` with a single "Open Folder" button
3. User clicks → native OS folder picker dialog opens (main process)
4. Selected path saved to `electron-store`, returned to renderer
5. File tree populates; user clicks a file to read it

**Returning launch:**
1. App opens → last folder path found in `electron-store`
2. File tree loads automatically — no prompts, ready instantly
3. User can change folder any time via **File → Open Folder** (menu bar) or the toolbar button

## Markdown Pipeline

Built once at module load in `markdown-pipeline.ts`, reused for every render:

```
remark-parse
  → remark-frontmatter    (parses YAML --- blocks; extracted separately for FrontmatterPanel)
  → remark-gfm            (tables, strikethrough, task lists)
  → remark-math           ($$…$$ and $…$ syntax)
  → remark-rehype         (converts MDAST → HAST)
  → rehype-highlight      (syntax highlighting via highlight.js)
  → rehype-katex          (renders math nodes to HTML)
  → callouts-plugin       (blockquote [!NOTE/WARNING/TIP/IMPORTANT] → styled divs)
  → rehype-stringify      (HAST → HTML string)
```

**Mermaid** is handled post-render: after React sets `innerHTML`, `mermaid.initialize()` scans for `<code class="language-mermaid">` blocks and replaces them with rendered SVG in-place.

**Images** resolve relative paths against the open folder using Electron's `file://` protocol — no extra plugin needed.

| macOS | `.dmg` | Drag-to-Applications; signed for Gatekeeper |

**Scripts:**
- `npm run dev` — hot-reload dev mode (electron-vite)
- `npm run build` — production build + installer output to `dist/`

Preferences are stored in the OS standard app-data location:
- Windows: `%APPDATA%\MDViewer\`
- macOS: `~/Library/Application Support/MDViewer/`
- Linux: `~/.config/MDViewer/`

## Search (Nice-to-Have)

Triggered by `Cmd/Ctrl+P`. Opens `SearchOverlay` as a modal:
- On folder open, indexes all `.md` filenames and first 500 chars of each file in memory
- Fuzzy matches against filename and content preview as user types
- Selecting a result navigates the tree and opens the file
- Index refreshes on `fs:watch-event`

No external search library required — a simple in-memory filter over the already-loaded tree is sufficient for typical agentic workflow output sizes.

## Security

- `contextIsolation: true`, `nodeIntegration: false` in the renderer — standard Electron security posture
- All file system access goes through the main process via IPC; the renderer never touches `fs` directly
- Markdown is rendered as trusted HTML from local files — no remote content fetched, no XSS surface from external sources
